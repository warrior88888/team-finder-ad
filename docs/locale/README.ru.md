## О проекте

**TeamFinder** — это цифровая экосистема для совместной работы, созданная для того, чтобы помочь разработчикам,
дизайнерам и другим специалистам превращать идеи пет-проектов в реальность. Платформа обеспечивает удобный поиск
проектов для участия или команды для реализации собственных идей.

> Данная версия проекта является итоговой дипломной работой и полностью реализует **Вариант 1** технического задания —
> Система избранного и расширенная фильтрация.

## Технические особенности

**TeamFinder** построен с использованием лучших современных практик для удобства разработки, оптимизации и максимальной
безопасности.

<details>
<summary><strong>🏗️ Современная структура</strong></summary>
<br>

- **Архитектура** — современная архитектура Django-приложения с использованием паттерна разделения логики на сервисы и
  селекторы.
- **Сервисы** — изолированная бизнес-логика. Удобно переиспользовать в разных приложениях, редактировать и настраивать.
- **Селекторы** — получение информации. Легко использовать в разных приложениях. Удобно оптимизировать и редактировать.
- **Core** — общая логика работы приложения, решения по инфраструктуре, безопасности и предоставлению единого интерфейса
  для использования в разных приложениях.
- **Миддлвари** — кастомные миддлвари безопасности, метрик и логики.

</details>

<details>
<summary><strong>🔒 Строгая типизация</strong></summary>
<br>

- **Типы** — код строго типизирован и адаптирован для Django с помощью `django-stubs`. Импорты типов оптимизированы с
  помощью `typing.TYPE_CHECKING`.
- **Проверка** — `pyright` осуществляет статическую проверку, настроен для `Django`.
- **Pydantic** — `pydantic-settings` для основного конфига. Является единственным источником правды для всего проекта,
  обеспечивает безопасность, жёсткую типизацию и валидацию. На основе `pydantic.BaseModel` реализована валидация
  настроек сервиса.

</details>

<details>
<summary><strong>⚡ Высокая производительность</strong></summary>
<br>

- **Оптимизированная работа с БД** — минимальное количество запросов для каждой страницы или действия. Часть логики
  вынесена на сторону базы данных для скорости. `Debug-toolbar` подключается при локальной разработке в режиме `DEBUG`.
  **Менее 5 (1–4) запросов к БД на каждую страницу/эндпоинт.**
- **Redis** — широкое использование для менеджмента сессий, ограничения частоты запросов, защиты от брутфорса, метрик.

</details>

<details>
<summary><strong>🛠️ Опыт разработчика</strong></summary>
<br>

- **uv** — быстрое, безопасное и эффективное управление зависимостями при помощи `uv.lock`. Является стандартом
  индустрии и в десятки раз ускоряет скорость развёртывания проекта и сборки образа.
- **Ruff** — максимально быстрое форматирование и проверка кода. Инструмент полностью настроен для Django-проекта в
  `pyproject.toml`. Форматирование в стиле `Black`.
- **Тестирование** — полное покрытие кода тестами с использованием `pytest`, `unittest`. Тесты полностью отделены от
  реальных баз `Redis` и `PostgreSQL`, в `Redis` используют выделенную базу данных, не оставляют после себя никаких
  файлов и записей, могут запускаться без переменных окружения. Использование плагинов, фабрик и `sqlite3 in-memory`
  обеспечивает максимальную скорость и чистый код. **210+ тестов выполняются менее чем за 5 секунд при первом запуске.**

</details>

<details>
<summary><strong>🛡️ Безопасность</strong></summary>
<br>

- **Брутфорс-защита** — собственный механизм ограничения частоты запросов и защиты от брутфорса. Потребляет мало
  ресурсов благодаря использованию `Redis`. Быстрая и гибкая настройка в `team_finder/settings.py`.
- **Nginx** — широко настроенная конфигурация даёт максимальную защиту от ботов, сканеров, DDoS. Более 90% стандартных
  бот-запросов получают ответ `444` (мгновенный разрыв соединения). Плохие запросы не доходят до Django. Реализовано
  кэширование медиа и статики на стороне клиента; клиент всегда получает актуальные файлы, так как каждый файл получает
  хэш при создании медиа и при выполнении `collectstatic`.
- **HTTPS** — работа с SSL-сертификатами, актуальные шифры, кэширование сессий.
- **Сложные пути** — возможность установить нестандартные пути для админ-панели и healthcheck.

</details>

<details>
<summary><strong>📋 Логирование</strong></summary>
<br>

- **Информативность** — логирование всех пользовательских действий. Информативные форматированные логи.
- **Лёгкий доступ** — логи доступны как файлы Docker volumes и логи контейнеров. Для уровней логирования созданы
  отдельные файлы.
- **Безопасность** — PK является уникальным идентификатором в логах пользовательских действий. Это обеспечивает защиту
  от утечек персональных данных (email, телефон), а также гарантирует неизменяемость уникального идентификатора.
- **Кастомизация** — можно настроить формат и уровень логов. Предусмотрена возможность установить разные уровни логов
  Django-приложения и базы данных.

</details>

<details>
<summary><strong>🚀 Production Ready</strong></summary>
<br>

- **Автоматизированный CI/CD** — Pipeline `GitHub Actions` по схеме: Тестирование → Сборка Docker-образа и пуш на
  `DockerHub` → Копирование конфигов на сервер → Перезапуск приложения. Активируется при каждом пуше в ветку `main` за
  исключением коммитов документации. Безопасное хранение секретов в `Actions secrets`. Используется кэширование для
  ускорения выполнения.
- **Dockerfile** — быстрый, лёгкий (295 MB) и защищённый образ. Использованы лучшие практики: `uv`, кэширование,
  компиляция кода на этапе сборки; базовый слой на `Debian` для корректной работы сложных библиотек. Основной процесс
  запускается от пользователя `django`, за корректный запуск и решение проблем с правами отвечает `gosu`.
- **Docker Compose** — актуальные `Alpine` версии сервисов. Созданы закрытые сети, доступ снаружи открыт только к
  `Nginx`. Сервисы максимально настроены для безопасности и оптимизации.

</details>

<details>
<summary><strong>🎨 Управление</strong></summary>
<br>

- **Jazzmin** — красивая и информативная админ-панель уровня современных дашбордов. Реализовано отображение кастомной
  метрики количества онлайн-пользователей. Подробнее — [демо](../images/admin-ui.png).
- **Действия** — кастомные действия в админке, доступна широкая функциональность и полное управление сайтом.
- **Фильтры** — большое количество фильтров, в том числе кастомный фильтр для выборки пользователей с онлайн-статусом.
- **Команды Django** — получить безопасные сервисные ссылки через `get_service_links`, сгенерировать аватарки по выборке
  пользователей через `generate_avatars` с использованием флагов.

</details>

<details>
<summary><strong>🎛️ Оптимизация</strong></summary>
<br>

- **Сервер** — образ собран для `Ubuntu 24.04`. Минимальные требования: 1 CPU, 2 GB ОЗУ, 2 GB HDD.
- **Легковесность** — на `Ubuntu 24.04` приложение занимает всего 650 MB виртуального веса `Docker` (около 900 MB
  физически).
- **Минимальное использование ОЗУ** — расход всего приложения с тремя воркерами `gunicorn`, запущенной системой и
  `docker` не превышает 1.2 GB RAM. Подробнее — [статистика](../images/vps-stat.png).

</details>

## Локальная разработка

### Клонируйте репозиторий

```bash
git clone git@github.com:warrior88888/team-finder-ad.git
cd team-finder-ad
```

### Быстрый старт (Docker)

Рекомендуемый способ — запуск через `docker compose`, он поднимает все сервисы автоматически:

```bash
docker compose -f docker-compose.dev.yml up --build
```

---

### Ручная установка

#### Требования

| Сервис               | Ссылка                            |
|:---------------------|:----------------------------------|
| Python 3.13          | https://www.python.org/downloads/ |
| uv *(рекомендуется)* | https://docs.astral.sh/uv/        |
| PostgreSQL           | https://www.postgresql.org/       |
| Redis                | https://redis.io/                 |

#### 1. Установка зависимостей

**С uv (рекомендуется):**

```bash
uv sync
```

**С pip:**

macOS / Linux:

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

Windows:

```bash
python -m venv .venv && .venv\Scripts\activate
pip install -r requirements.txt
```

#### 2. Настройка окружения

```bash
cp .env.example .env
```

Откройте `.env` и заполните по инструкции внутри файла.

#### 3. Миграции

```bash
python manage.py migrate
```

#### 4. Запуск сервера

```bash
python manage.py runserver
```

#### 5. Демо-данные *(опционально)*

Создаёт тестовых пользователей с проектами — рекомендуется для локальной разработки:

```bash
python manage.py loaddata demo_data.json
python manage.py generate_avatars --blank
```

> Все пользователи из `demo_data.json` имеют пароль: **`devpass`**

#### 6. Служебные ссылки *(опционально)*

Выводит ссылки для быстрого перехода в админ-панель и healthcheck:

```bash
python manage.py get_service_links
```

---

### Качество кода

```bash
ruff check . --fix   # линтинг
pyright              # статическая типизация
pytest               # тесты
```

---

### Примечания

- `requirements.txt` содержит **все** зависимости, включая dev.
- `pytest` не требует `.env` файла, но нужен запущенный Redis.
- В контейнере `django_app` установлен **uv**, команды можно вызывать без `uv run`:

```bash
docker exec django_app python manage.py <команда>
```

## Развертывание в продакшене

Развертывание полностью автоматизировано с использованием **GitHub Actions**. Каждый пуш в ветку `main` (за исключением
коммитов документации) запускает пайплайн:

```
Tests → Build → Deploy → Notify
```

### Процесс CI/CD Pipeline

|           Шаг            | Описание                                                                                                                                                             |
|:------------------------:|:---------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **1. Актуальные версии** | Используются последние версии actions на базе **Node.js 24** для устранения предупреждений.                                                                          |
|   **2. Тестирование**    | Настройка окружения с **Python 3.13** и **uv**. Запускает **Ruff** (линтинг), **Pyright** (статическая типизация) и **Pytest** с живым сервис-контейнером **Redis**. |
|  **3. Контейнеризация**  | Сборка оптимизированного продакшн-образа через **Docker Buildx** с кэшированием GHA и публикацией на **DockerHub**.                                                  |
|      **4. Деплой**       | Копирование конфигураций на VPS → генерация `.env` из GitHub Secrets → `docker compose pull && up -d` → очистка старых образов.                                      |
|    **5. Уведомления**    | Отправка отчётов об успехе/ошибке с сообщениями коммитов и информацией об авторе прямо в **Telegram**-бот.                                                           |
